*Write the simple spec first. Let the implementation be whatever Go you want. Gate the repo on pointwise equivalence.*

---

`shen-derive` is the sibling to `shen-guard`.

Use `shen-guard` when the problem is **I/O, validation, mutation, and glue code** — wherever you want the Go compiler to enforce a domain invariant via opaque constructors.

Use `shen-derive` when the problem is **a pure function whose obvious spec is clear but whose efficient implementation is not** — and you want the repo to notice the moment they drift apart.

The mental model is simple:

1. Write the computation in Shen as the obvious-correct `(define ...)` spec.
2. A human or an LLM writes the Go implementation however they like.
3. `shen-derive verify` evaluates the spec on sampled inputs and emits a table-driven Go test that asserts the implementation matches, pointwise.
4. Commit the generated test. Gate the repo on two things: (a) regenerating the test from the current spec produces the same file (no drift), and (b) running the test passes (no mismatch).

The spec is the oracle. The Go is free-form. The gate catches you when you lie.

## What changed from v1

An earlier version of `shen-derive` tried to *generate* the fast Go from algebraic rewrites of the slow spec — Bird-Meertens fusion laws, pattern-based lowering to for-loops, the whole workbench. It worked. It also hit a ceiling: every new shape of computation needed a new law *and* a new lowering *and* a fresh equivalence proof against the naive form.

The verification-gate model flips the polarity. Instead of generating the fast code from the slow spec, trust the author (LLM or human) to write the fast code and use the slow spec as the oracle that catches mistakes. The set of specs `shen-derive` can verify is now *any pure functional computation Shen can express*, which is a strictly larger space than the rewrite engine ever reached.

The old engine is archived as `.go.bak` under `shen-derive/archive/`. It may return as an *LLM hint* ("this spec fuses to a single pass"), but not as a code generator.

## The payment example

The flagship is the balance-never-negative invariant in `examples/payment/`. The spec in `specs/core.shen`:

```shen
(define processable
  {amount --> (list transaction) --> boolean}
  B0 Txs -> (foldr (lambda X (lambda Acc (and (>= (val X) 0) Acc)))
              true
              (scanl (lambda B (lambda Tx (- (val B) (val (amount Tx))))) (val B0) Txs)))
```

Read it literally:

- start with the initial balance `B0`
- apply each transaction in order, keeping the running balances with `scanl`
- check that every running balance stays non-negative

That's easy to trust. It's also slow: it allocates an intermediate list of running balances and walks it twice.

The hand-written Go in `internal/derived/processable.go` is the loop you'd actually ship:

```go
func Processable(b0 shenguard.Amount, txs []shenguard.Transaction) bool {
    balance := b0.Val()
    for _, tx := range txs {
        balance -= tx.Amount().Val()
        if balance < 0 {
            return false
        }
    }
    return true
}
```

No intermediate list, early exit, single pass. Nothing about this code looks like the spec. That's fine. That's the point.

`shen-derive verify` bridges the gap by generating `internal/derived/processable_spec_test.go`:

```go
// Code generated by shen-derive. DO NOT EDIT.
// Regenerate with: shen-derive verify <spec.shen> --func processable
//
// This file checks that Processable matches the Shen spec
// by evaluating the spec on sampled inputs and comparing outputs.

package derived_test

// ... mustAmount / mustTransaction / mustAccountId helpers ...

func TestSpec_Processable(t *testing.T) {
    cases := []struct {
        name string
        b0   shenguard.Amount
        txs  []shenguard.Transaction
        want bool
    }{
        {
            name: "case_00",
            b0:   mustAmount(0),
            txs:  []shenguard.Transaction{},
            want: true,
        },
        // ... 34 more cases ...
    }
    for _, tc := range cases {
        t.Run(tc.name, func(t *testing.T) {
            got := derived.Processable(tc.b0, tc.txs)
            if got != tc.want {
                t.Fatalf("%s: spec says %v, impl returned %v", tc.name, tc.want, got)
            }
        })
    }
}
```

Thirty-five table-driven cases, each with inputs drawn from the boundary pool (`0, 1, -1, 5, 2.5, 100` for numbers) combined across the list-of-transactions shapes (empty, singletons, small multi-element). The `want` column comes from the spec evaluator running over the same inputs.

Break the implementation — say, flip `balance -= tx.Amount().Val()` to `+=` — and `go test` lights up with six failing cases like:

```
--- FAIL: TestSpec_Processable/case_13
    processable_spec_test.go:243: case_13: spec says false, impl returned true
```

Break the spec, or change the sampling strategy, and the drift gate fires before the test even runs.

## How the harness thinks about samples

For a spec like `processable` the harness walks the type signature:

- `amount` is a wrapper around `number` with a `(>= X 0)` `verified` predicate. The number pool is `{0, 1, -1, 5, 2.5, 100}`; the predicate drops the negative so `amount` samples become `{0, 1, 5, 2.5, 100}`.
- `transaction` is a composite `[Amount From To]`. The harness builds one variation per field-sample index, which gives five transactions, each with a distinct amount.
- `(list transaction)` sampling produces the empty list plus one singleton per transaction sample (capped at 6) plus a three-element mix. That's roughly eight list shapes.
- The cartesian product of five balances × eight list shapes is 40, capped at `--max-cases 50`, yielding the 35-case table.

The fractional amount (`2.5`) matters. An earlier version of the sampler bundled only three composites and the tricky fractional one never ended up inside any list, so an implementation that silently `int`-truncated amounts passed unnoticed. The current "one singleton per elem sample" rule is what surfaces that bug.

### Seeded random sampling

For deeper exploration you can layer seeded random draws on top of the boundary set:

```bash
shen-derive verify specs/core.shen \
  --func processable \
  --impl-pkg your-module/internal/derived \
  --impl-func Processable \
  --import your-module/internal/shenguard \
  --seed 42 --random-draws 16 \
  --out your/internal/derived/processable_spec_test.go
```

With `--seed 42`, the number pool becomes the six boundary values plus sixteen random draws (half ints in `[-1000, 1000]`, half two-decimal floats in the same range). The string pool gets sixteen random alphanumeric strings of length 1-8. The seed is stamped into the generated file's header so runs are reproducible.

Crucially, the default is `seed=0` — deterministic, boundary-only. Random sampling is opt-in because the committed test file is part of the gate; silent non-determinism would turn into CI churn.

## Multi-clause defines with `where` guards

The payment spec is a single-clause define. Real specs often need pattern matching. Take `pair-in-list?` from `examples/dosage-calculator/specs/core.shen`:

```shen
(define pair-in-list?
  _ _ [] -> false
  A B [[X Y] | Rest] -> true  where (and (= A X) (= B Y))
  A B [[X Y] | Rest] -> true  where (and (= A Y) (= B X))
  A B [_ | Rest] -> (pair-in-list? A B Rest))
```

Four clauses. The first fires on the empty list. The second and third have `where` guards that compare the pair against the head of the list in both orders. The fourth recurses on the tail.

`shen-derive` parses this into four `Clause` records with patterns, guards, and bodies all stored as s-expressions. The evaluator dispatches clause-by-clause: try each pattern, evaluate the guard if present, run the first body that matches. Recursion works because the harness binds every define in the spec file as a curried `BuiltinFn` in the base evaluation environment, so `(pair-in-list? A B Rest)` inside clause 4 resolves exactly the way `(foldr ...)` resolves inside `processable`.

The pattern matcher is small enough to describe in a paragraph: wildcards (`_`) bind nothing, uppercase atoms (`A`, `B`, `Rest`) bind to the value, the `nil` atom matches the empty list, `cons` heads match non-empty lists by destructuring head and tail recursively, and fixed-length list patterns like `[X Y]` work automatically via their desugaring to `(cons X (cons Y nil))`.

That's the whole pattern-matching story. Nine tests cover it in `shen-derive/core/match_test.go`.

## The `sb derive` gate

`sb derive` is an additional gate `sb` registers automatically when the project has configured any `[[derive.specs]]` entries. Add one to `sb.toml`:

```toml
[derive]
dir = "../../shen-derive"

[[derive.specs]]
path      = "specs/core.shen"
func      = "processable"
impl_pkg  = "ralph-shen-agent/internal/derived"
impl_func = "Processable"
guard_pkg = "ralph-shen-agent/internal/shenguard"
out_file  = "internal/derived/processable_spec_test.go"
```

Then `sb derive` (or `sb gates`) runs `shen-derive verify` via `go run`, diffs the regenerated test against the committed `out_file`, fails on drift with a unified diff, and runs `go test` on each referenced impl package. `sb derive --regen` rewrites the committed file in place when the drift is intentional.

The `cmd/sb` module and the `shen-derive` module are separate Go modules by design, so the integration is subprocess-based, matching how `examples/payment/Makefile` has done it all along — `sb derive` just lifts that Makefile pattern into the CLI so every project can opt in with config alone.

You can configure whichever subset of gates you actually want. In the payment example, `make all` runs `go build`, `go test`, `shen-check.sh`, and `make shen-derive-verify`. `sb gates` runs the fuller pipeline including `shengen` regeneration and a TCB audit. Neither is more canonical than the other — pick the set that matches what you want the repo to guarantee.

## The honest story about what this proves

`shen-derive verify` is **property-style sampling with a deterministic default**, not mathematical proof. The gate catches:

- implementation bugs that disagree with the spec on any sampled input
- spec changes that drift the generated test file
- sampling-strategy changes that invalidate committed results

The gate does **not** prove:

- that the implementation matches the spec on *all* inputs
- that the spec is itself correct
- anything about inputs outside the sampled space

That's fine for most engineering contexts. The sequent-calculus side of Shen-Backpressure still handles the "for all inputs" claims via `shen-check` and the guard-type constructors — `shen-derive` is the complementary tool for "the Go loop matches the obvious spec on the cases we checked, and the repo notices the moment that stops being true."

If you want stronger guarantees on a specific function, pair shen-derive's sampling with Go's native fuzzer using the spec evaluator as the oracle — that's a natural next step the harness is already shaped for.

## Where to start

If you want one place to look, use the payment example:

```bash
cd examples/payment
make shen-derive-verify
# wrote /tmp/shen-derive-XXX.go (35 cases)
# ok  	ralph-shen-agent/internal/derived
```

Then break the impl — flip a sign, drop an early return — and watch the gate fire. Then run `make shen-derive-regen` after an intentional spec change and see the diff land.

That's `shen-derive` in one snapshot:

- an obvious Shen spec
- a free-form Go implementation
- a committed test file generated from the first, checking the second
- a gate that refuses to let them drift apart

And because the spec is normal Shen, `shen-guard` and `shengen` happily share the same file. The two tools are complementary, not layered: pick the one that fits the shape of the function you're pinning.
