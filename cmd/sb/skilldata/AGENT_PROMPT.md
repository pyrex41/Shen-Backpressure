# Shen-Backpressure Agent Reference

You are operating inside a Ralph loop with Shen sequent-calculus backpressure and a codegen bridge. Every iteration, your changes must pass the configured gates before advancing: the core five, plus `shen-derive` when the project has derive coverage configured.

## The Core Five Gates

```
Gate 1: shengen      → Regenerates guard types from specs/core.shen
Gate 2: test         → Tests run against the regenerated types
Gate 3: build        → Compilation/type check against regenerated types
Gate 4: shen tc+     → Verifies the spec is internally consistent
Gate 5: tcb audit    → Diffs generated code, rejects unexpected files in shenguard/
```

Gate 1 runs first. If you change `specs/core.shen`, the generated guard types change. If your code uses old type signatures, Gate 3 fails. If your tests pass wrong values to constructors, Gate 2 fails. If your spec is contradictory, Gate 4 fails. If you hand-edit generated files or add non-generated files to the guard types directory, Gate 5 fails.

## Optional Gate 6: shen-derive

If `sb.toml` contains any `[[derive.specs]]` entries, the loop also runs:

```
Gate 6: shen-derive → Regenerates committed spec-derived tests, fails on drift, then runs go test on referenced impl packages
```

When this gate is present, changing a `(define ...)` spec, sampling seed, or handwritten implementation can cause drift. Run `sb derive --regen` to refresh the committed generated test when the drift is intentional.

## Live Project Context

The `sb context` command emits the current project's guard types, gate configuration, derive coverage, and backpressure state. The Ralph loop injects this output into your prompt each iteration under the "## Live Project Context" heading. Trust it as the source of truth for:

- Available guard types, their categories, and their constructor signatures
- The current proof chain (type dependencies)
- Configured gates and their execution order
- Latest backpressure failure, if any

Do not infer or memorize project-specific facts — they live in the context block, which is always fresh. If the context block is missing or says "(context unavailable)", fall back to reading `specs/core.shen` and `sb.toml` directly.

## Context Files — Read Every Iteration

- `specs/core.shen` — Shen sequent-calculus type rules (the formal spec)
- The generated guard types file (DO NOT EDIT — regenerated by Gate 1)
- `sb.toml` — Especially important when `[[derive.specs]]` is configured
- `plans/fix_plan.md` — Task plan with `- [ ]` items
- `prompts/main_prompt.md` — May contain backpressure errors from the previous iteration

## Guard Type Discipline

The generated types have **private/unexported fields** — the only way to create them is through their constructors. This is the bridge between Shen's formal proofs and the target language's type system. The **compiler** enforces this, not you — if you skip a step, Gate 3 fails.

### Rule 1: Wrap at the Boundary

At entry points (HTTP handlers, CLI commands, message consumers), immediately wrap raw values in guard types. The constructor validates the spec's constraints and returns an error if they're violated. Consult the live context block for the exact constructor names and signatures in this project.

### Rule 2: Trust Internally

Once values are wrapped in guard types, trust them. Internal functions accept and return guard types, not primitives. Don't re-validate what the constructor already checked.

### Rule 3: Follow the Proof Chain

If type B requires type A as a parameter, you must construct A first. The compiler enforces this — you cannot construct B without first having a valid A. The live context block emits this project's specific proof chain; follow it step by step. You cannot skip a step.

### Rule 4: Extract with Accessors

To get raw values back (for SQL queries, JSON output, templates), use the accessor methods. Fields are private — direct field access won't compile.

## Why the Compiler Catches It

The trick is private fields. The only path to construct a guard type is its constructor. Writing a struct literal with private fields won't compile in Go, Rust, or TypeScript. The compiler — not you — enforces that every guard type in scope has passed through its proof obligations.

## Anti-Patterns

**Never edit the generated guard types file** — it's regenerated every iteration. Change `specs/core.shen` instead.

**Never construct guard types by bypassing the constructor** — the compiler will reject it (fields are private).

**Never ignore constructor errors** — if the constructor returns an error, handle it. The error means the spec's invariant is violated.

**Never duplicate validation** — the constructor already validates. Don't check `amount >= 0` in your handler if the constructor already does it.

## When a Gate Fails

| Gate | Failure means | Fix |
|------|--------------|-----|
| shengen | Spec has syntax errors | Fix `specs/core.shen` |
| test | Constructor rejecting test input, or handler not using guard types | Fix test data or handler code |
| build | Generated types changed (spec evolved) and code uses old signatures | Update code to match new type signatures |
| shen tc+ | Spec is internally inconsistent | Fix contradictory rules in `specs/core.shen` |
| tcb audit | Generated file was hand-edited, or unexpected files in shenguard/ | Regenerate with shengen; remove non-generated files from shenguard/ |
| shen-derive | Generated spec-derived test drifted, or the impl no longer matches the sampled spec cases | Review the diff, run `sb derive --regen` if intentional, or fix the implementation/spec mismatch |

## Iteration Rules

1. Pick the FIRST unchecked `- [ ]` item from `plans/fix_plan.md`. Implement it fully. Check it off. Stop.
2. If backpressure errors appear below, fix those FIRST before working on plan items.
3. Every new domain concept needs a datatype in `specs/core.shen`. Add the spec first, then write application code.
4. No placeholders. Every file must be complete, compilable, runnable.
5. All configured gates must pass before you stop. If derive coverage is enabled, that includes the `shen-derive` drift gate.

## Hardened Mode

If the project uses `--mode hardened`, the generated guard types have additional bypass prevention (sealed interfaces, zero-value traps, non-clonable proofs, `#private` fields, closure vaults, HMAC provenance). The live context block will note when hardened mode is active. When writing code against hardened types, always go through the public factory/constructor API — never try to name concrete types, clone proofs, or cast around private fields.
