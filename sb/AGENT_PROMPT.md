# Shen-Backpressure Agent Reference

You are operating inside a Ralph loop with Shen sequent-calculus backpressure and a codegen bridge. Every iteration, your changes must pass four gates before advancing.

## The Four Gates

```
Gate 1: shengen      → Regenerates guard types from specs/core.shen
Gate 2: test         → Tests run against the regenerated types
Gate 3: build        → Compilation/type check against regenerated types
Gate 4: shen tc+     → Verifies the spec is internally consistent
```

Gate 1 runs first. If you change `specs/core.shen`, the generated guard types change. If your code uses old type signatures, Gate 3 fails. If your tests pass wrong values to constructors, Gate 2 fails. If your spec is contradictory, Gate 4 fails.

## Context Files — Read Every Iteration

- `specs/core.shen` — Shen sequent-calculus type rules (the formal spec)
- The generated guard types file (DO NOT EDIT — regenerated by Gate 1)
- `plans/fix_plan.md` — Task plan with `- [ ]` items
- `prompts/main_prompt.md` — May contain backpressure errors from the previous iteration

## Guard Type Discipline

The generated types have **private/unexported fields** — the only way to create them is through their constructors. This is the bridge between Shen's formal proofs and the target language's type system. The **compiler** enforces this, not you — if you skip a step, Gate 3 fails.

### Rule 1: Wrap at the Boundary

At entry points (HTTP handlers, CLI commands, message consumers), immediately wrap raw values in guard types. The constructor validates the spec's constraints and returns an error if they're violated.

### Rule 2: Trust Internally

Once values are wrapped in guard types, trust them. Internal functions accept and return guard types, not primitives. Don't re-validate what the constructor already checked.

### Rule 3: Follow the Proof Chain

If type B requires type A as a parameter, you must construct A first. The compiler enforces this — you cannot construct B without first having a valid A. Example:

```
Raw input → NewAmount(raw) → Amount
Amount + AccountIds → NewTransaction(Amount, AccountId, AccountId) → Transaction
Balance + Transaction → NewBalanceChecked(balance, Transaction) → BalanceChecked
Transaction + BalanceChecked → NewSafeTransfer(Transaction, BalanceChecked) → SafeTransfer
```

You cannot skip a step. SafeTransfer requires a BalanceChecked, which requires proving the balance covers the transaction.

### Rule 4: Extract with Accessors

To get raw values back (for SQL queries, JSON output, templates), use the accessor methods on guard types. Fields are private — direct field access won't compile.

## Spec Patterns

### Wrapper (no validation)
```shen
(datatype account-id
  X : string;
  ==============
  X : account-id;)
```
→ Constructor takes a string, returns the guard type. No validation, no error.

### Constrained (validated)
```shen
(datatype amount
  X : number;
  (>= X 0) : verified;
  ====================
  X : amount;)
```
→ Constructor takes a number, returns (guard type, error). Rejects negative values.

### Composite (struct)
```shen
(datatype transaction
  Amount : amount;
  From : account-id;
  To : account-id;
  ===================================
  [Amount From To] : transaction;)
```
→ Constructor takes already-validated guard types as fields. No error — inputs are pre-validated.

### Guarded (invariant proof)
```shen
(datatype balance-invariant
  Bal : number;
  Tx : transaction;
  (>= Bal (head Tx)) : verified;
  =======================================
  [Bal Tx] : balance-checked;)
```
→ Constructor takes a number and a Transaction, checks that balance >= transaction amount. Returns error if invariant is violated.

## Anti-Patterns

**Never edit the generated guard types file** — it's regenerated every iteration. Change `specs/core.shen` instead.

**Never construct guard types by bypassing the constructor** — the compiler will reject it (fields are private).

**Never ignore constructor errors** — if the constructor returns an error, handle it. The error means the spec's invariant is violated.

**Never duplicate validation** — the constructor already validates. Don't check `amount >= 0` in your handler if the constructor already does it.

## When a Gate Fails

| Gate | Failure means | Fix |
|------|--------------|-----|
| shengen | Spec has syntax errors | Fix `specs/core.shen` |
| test | Constructor rejecting test input, or handler not using guard types | Fix test data or handler code |
| build | Generated types changed (spec evolved) and code uses old signatures | Update code to match new type signatures |
| shen tc+ | Spec is internally inconsistent | Fix contradictory rules in `specs/core.shen` |
