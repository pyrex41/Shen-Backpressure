# Multi-Tenant SaaS API — Shen-Backpressure Agent Prompt

You are operating inside a Ralph loop with Shen sequent-calculus backpressure and a codegen bridge. Every iteration, your changes must pass four gates before advancing.

## Domain

Multi-tenant SaaS API in Go where every data access carries proof of authentication and tenant-scoped authorization. The proof chain is: JWT validation → AuthenticatedUser → TenantAccess → ResourceAccess. You cannot construct a ResourceAccess without proving tenant membership first — the Go compiler enforces this through shengen guard types.

Stack: Go stdlib net/http, SQLite, htmx frontend for an admin dashboard. No frameworks.

## The Four Gates

```
Gate 1: shengen      → Regenerates Go guard types from specs/core.shen
Gate 2: go test      → Tests run against the regenerated types
Gate 3: go build     → Compilation check (builds all packages)
Gate 4: shen tc+     → Verifies the spec is internally consistent
```

Gate 1 runs first. If you change `specs/core.shen`, the generated types in `internal/shenguard/guards_gen.go` change. If your Go code uses old type signatures, Gate 3 fails. If your tests pass wrong values to constructors, Gate 2 fails. If your spec is contradictory, Gate 4 fails.

## Context Files — Read Every Iteration

- `specs/core.shen` — Shen sequent-calculus type rules (the formal spec)
- `internal/shenguard/guards_gen.go` — Generated guard types (DO NOT EDIT — regenerated by Gate 1)
- `plans/fix_plan.md` — Task plan with `- [ ]` items

## Domain Entities

- **Users** with JWT tokens and tenant memberships
- **Tenants** with isolated data
- **Resources** (documents) owned by tenants
- **API endpoints** that require authorization proofs

## Invariants (enforced by Shen specs + guard types)

- An authenticated user requires a non-expired JWT token
- Tenant access requires an authenticated user who is a member of that tenant
- Resource access requires tenant access where the tenant owns the resource
- No endpoint can return data without a ResourceAccess proof
- Cross-tenant data access is impossible by construction

## Guard Type Discipline

The generated types in `internal/shenguard/` have **unexported fields** — the only way to create them is through their constructors. This is the bridge between Shen's formal proofs and Go's type system.

### Rule 1: Wrap at the Boundary

At entry points (HTTP handlers), immediately wrap raw values in guard types:

```go
import "multi-tenant-api/internal/shenguard"

token, err := shenguard.NewJwtToken(rawToken)
if err != nil {
    http.Error(w, "invalid token", 401)
    return
}
```

### Rule 2: Trust Internally

Once values are wrapped in guard types, trust them. Internal functions accept and return guard types, not primitives:

```go
// GOOD: accepts guard type
func getResources(access shenguard.TenantAccess) ([]Resource, error) { ... }

// BAD: accepts raw primitives
func getResources(tenantID string, userID string) ([]Resource, error) { ... }
```

### Rule 3: Follow the Proof Chain

ResourceAccess requires building up proofs step by step:

```
Raw token → NewJwtToken(string) → JwtToken
Expiry check → NewTokenExpiry(exp, now) → (TokenExpiry, error)
JWT + Expiry + UserID → NewAuthenticatedUser(JwtToken, TokenExpiry, UserId) → AuthenticatedUser
Auth + TenantID + membership check → NewTenantAccess(AuthenticatedUser, TenantId, bool) → (TenantAccess, error)
TenantAccess + ResourceID + ownership check → NewResourceAccess(TenantAccess, ResourceId, bool) → (ResourceAccess, error)
```

You cannot skip a step. ResourceAccess requires TenantAccess, which requires AuthenticatedUser, which requires JwtToken + TokenExpiry.

### Rule 4: Extract with Accessor Methods

To get raw values back (for SQL queries, JSON output, templates):

```go
tokenStr := token.Val()           // string
userId := auth.User()             // UserId
tenantId := access.Tenant()       // TenantId
resourceId := ra.Resource()       // ResourceId
```

## HTTP Endpoints

- `POST /auth/login` — validates credentials, issues JWT, returns AuthenticatedUser proof
- `GET  /tenants/:id/resources` — list resources, requires TenantAccess proof
- `GET  /tenants/:id/resources/:rid` — get single resource, requires ResourceAccess proof
- `POST /tenants/:id/resources` — create resource, requires TenantAccess proof
- `GET  /admin` — dashboard showing tenants, users, and access logs

## Anti-Patterns

**Never edit `guards_gen.go`** — it's regenerated every iteration. Change `specs/core.shen` instead.

**Never construct struct literals** — `JwtToken{v: "abc"}` won't compile (unexported field). Use `NewJwtToken("abc")`.

**Never ignore constructor errors** — if `NewX` returns an error, handle it. The error means the spec's invariant is violated.

**Never duplicate validation** — the constructor already validates. Don't check `token != ""` in your handler if `NewJwtToken` already does it.
